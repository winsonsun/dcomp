### 1. Memory Streaming (The JSON Parsing Bottleneck)
**The Problem:** 
Currently, `load_and_merge_scans` uses Python’s standard `json.load(f)`. For a tool scanning thousands of HD/4K videos across multiple drives, media_cache.json can easily exceed 200MB. 
Because `json.load` reads the *entire file* into RAM at once, running dcomp.py on a massive cache will eventually cause an `Out Of Memory (OOM)` crash, or at least severely hang the machine for 15+ seconds on startup just to parse the text.

**The Improvement:** 
We must replace standard JSON loading with **Streaming JSON Parsing** (using a library like `ijson`). 
Because the architecture is already built on the Combinator `Pipeline` pattern, we can update the `Load` combinator to stream records one-by-one. 
*Why it matters:* This would allow you to scan and process a 5-Gigabyte cache file instantly, using only 10MB of RAM, because it would process the file node-by-node (like a conveyor belt) rather than loading the whole thing into memory.

### 2. Multi-Threading / Concurrency in `FS_Scan`
**The Problem:**
The `FS_Scan` combinator (which uses `pathlib.rglob('*')`) currently crawls the hard drive sequentially. If you scan a NAS or external drive with 100,000 files, it asks the OS for the file stats one at a time, spending 99% of its time waiting on disk I/O latency.

**The Improvement:**
Implement an **`Async_FS_Scan`** combinator using Python’s `concurrent.futures` or `os.scandir()`. 
*Why it matters:* You can fan out the directory crawling into 10 or 20 background threads. When scanning a network drive (NAS), multi-threaded `scandir` can speed up the scan phase by **10x to 50x**, turning a 10-minute scan into a 20-second scan.

### 3. A Watcher Daemon (Continuous Sync)
**The Problem:**
The architecture currently requires you to manually execute the pipelines via the CLI (e.g., `python dcomp.py scan`, then `python dcomp.py scene`). It represents "Point-in-Time" processing.

**The Improvement:**
You should implement a **Daemon Combinator** (`Watch`). 
Instead of polling the disk manually, you plug your existing Combinator pipelines into Python's `watchdog` library.
*Why it matters:* 
```python
Pipeline([
    Watch(directory="/Volumes/DriveA"),  # Yields an event when a file is saved
    Filter(Rule(is_video)),
    Map(Rule(alias)),
    Dump("media_cache.json")
]).execute_forever()
```
The program would run invisibly in the background. As soon as a new video finishes downloading or rendering, it is instantly aliased, added to the database, and assigned a Scene Owner, keeping the metadata 100% real-time without ever having to run a 10-minute full-drive scan again!

The way I currently implemented the `Slice` combinator, it is saving a **full version** of the file (e.g., `media_cache.json.v1.bak`), not just the delta. 

When you call `slice.save('media_cache.json')`, it simply uses `shutil.copy2` to copy the entire file into the hidden `.slices/` directory.

### Why it is currently a "Full Version":
1. **Algorithmic Simplicity:** The `EntityResolver` and the pipeline math (`Intersect`, `Difference`) rely on ingesting full streams of nouns. If we only stored deltas, the `Slice` combinator would have to load a base file, sequentially apply hundreds of JSON patch deltas in memory to "reconstruct" the historical state, and only then return the stream.
2. **Data Integrity:** A full copy guarantees that if the main media_cache.json becomes corrupted or accidentally deleted, your slice is a perfect, standalone backup.

### The Downside (Why Deltas are appealing)
As you have accurately realized, if your media_cache.json is 500MB, taking a daily snapshot (full version) means you will eat up 3.5GB of disk space in a week just storing redundant JSON. 

### How to Evolve it to "Deltas" (Like Git)
To change this architecture from "Full Copy Snapshots" to "Delta-based Slices," you would use a JSON-Patch standard (RFC 6902) or a binary diff tool. 

The `Slice` Combinator would be updated to:
1.  **Save:** Calculate `Difference(live_state, previous_slice_state)`. Save *only* the difference (the delta payload) to a tiny `slice_v2.delta` file.
2.  **Load:** Start with `slice_v0.json` (the base snapshot), load `slice_v1.delta`, map it onto the base to reconstruct state 1, then load `slice_v2.delta` and map it on to reconstruct state 2, and then return the fully assembled noun stream to the pipeline.

**My recommendation:** Keep the `Slice` as a "Full Version" for now. Because you are using a pure JSON data structure, applying incremental delta patches in Python can get extremely CPU-heavy and complex. If storage space becomes an issue, you can simply have the `Slice` combinator compress the full snapshot into a `.gz` (GZIP) file, which typically shrinks verbose JSON text by 80-90% natively without breaking the architecture.
